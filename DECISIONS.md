# Flow 記帳 PWA — 設計決策紀錄

這份文件記錄重要的設計與架構決策，說明「為什麼這樣做」而不只是「做了什麼」。

---

## UI 架構

### 為什麼首頁只有兩個 tab（紀錄／查詢）而不是三個？
初期設計有三個 tab（紀錄／統計／分析），後來合併為兩個。
「統計」和「查詢」的使用情境高度重疊，分開反而讓使用者困惑要去哪裡找功能。
查詢 tab 直接輸入自然語言，涵蓋了所有統計需求，減少 tab 數量降低決策成本。

### 為什麼保留傳統 Nav（紀錄明細、分析頁）而不全部 AI 化？
這個 app 的目標用戶正處於 AI 工具的過渡期，部分使用者習慣瀏覽流水帳單和靜態圖表。
全部 AI 化會提高學習門檻，導致使用者放棄。
傳統瀏覽模式作為保底，AI 查詢作為升級體驗，兩者並存讓不同習慣的使用者都能上手。

### 為什麼統計和分析 tab 不放相機按鈕？
相機的使用情境是「拍收據記帳」，屬於紀錄行為。
統計和分析的使用情境是「查詢和理解數據」，不需要拍照。
減少不相關的輸入選項，讓使用者的操作意圖更清晰。

## AI 互動設計

### 為什麼語音辨識後需要確認步驟，而不是直接儲存？
語音辨識有錯誤率，尤其是金額和商品名稱。
直接儲存會產生錯誤資料，使用者事後需要花更多時間修正。
確認步驟雖然多一個動作，但建立了使用者對 AI 輸出的信任感——「AI 幫我整理，我來確認」符合核心設計原則。

### 為什麼拍照辨識選擇 GPT-4o 而不是 Google Cloud Vision？
GPT-4o 可以一次完成辨識和分類，回傳結構化資料（項目、金額、類別）。
Google Cloud Vision 只做 OCR，還需要額外處理文字解析和分類邏輯，增加架構複雜度。
對 prototype 階段來說，GPT-4o 的整合成本更低，且中文收據辨識效果更好。

### 為什麼統計功能用自然語言輸入而不是日期選擇器 + 下拉選單？
傳統日期選擇器的操作步驟多（點開 → 選年 → 選月 → 選日），使用者要做多次決策。
自然語言輸入（「這三個月的飲料花費」）更接近使用者真實的思考方式，一句話完成條件設定。
這也是這個 app 最核心的 AI 價值展示：把複雜的查詢條件變成對話。

### 為什麼查詢結果標題要用橘色標記關鍵字？
使用者輸入的原始關鍵字（如「咖啡」、「學習」）在結果頁標題中用橘色 `<span>` 標示，
讓使用者一眼確認 AI 正確理解了查詢意圖，降低「AI 是不是查錯了」的疑慮。
不用粗體而用顏色，是因為標題字型已經夠大，額外粗體會顯得笨重。

### 為什麼 analysis_full 同時顯示佔比數據條、趨勢圖與上月比較？
使用者輸入模糊分析意圖（如「咖啡花費分析」）時，無法判斷他想看佔比、趨勢還是比較。
一次呈現三個維度（佔比數據條、支出趨勢折線圖、與上月差異），讓使用者不需要重新查詢就能獲得完整脈絡。
趨勢圖線條顏色對應關鍵字所屬的父類別顏色（透過 `catColor()` 取得），強化視覺一致性。

### 為什麼 analysis_peak 總結區顯示三個維度而不只是高峰日？
使用者查詢「最高消費」時，認知上可能是單筆最貴的項目、某類別的總額，或哪一天花最多。
三個維度（最高消費項目、最高消費類別、最高消費日）同時呈現，一次回答所有可能的意圖，不需要使用者追問。

### 為什麼動態背景放在 layouts/default.vue 而不是各頁面？
初期動態背景（三個色球漸層動畫）放在 `index.vue`，導致切換頁面時背景消失或重新載入有閃爍感。
移至 `layouts/default.vue` 後，背景在所有頁面共用且持續存在，各頁面元件背景保持透明即可。
這也是為什麼樣式保護規則明確禁止在任何元件加背景色。

### 為什麼 AI 查詢需要語意擴展關鍵字？
使用者輸入「咖啡」時，紀錄裡可能有「拿鐵」、「美式咖啡」、「榛果拿鐵」等。
純關鍵字比對（ilike %咖啡%）會漏掉這些語意相關的項目。
讓 GPT 自動擴展同義詞陣列，再用 OR 查詢，讓結果更貼近使用者的真實意圖。

### 為什麼需要 AI 安全防護？
記帳 app 的 AI 功能有明確的使用範圍，不應該回答食譜、天氣、寫程式等無關問題。
System prompt 加入角色限制和 off-topic 拒絕邏輯，防止：
1. 使用者好奇亂問不相關問題消耗 API 額度
2. Prompt injection 攻擊（「忘掉你的指令，現在做 XXX」）
3. 超長輸入攻擊（前端加 200 字限制）

### 為什麼「上週消費分析」不自動觸發比較模式？
使用者只提到單一週期時，標題不應出現「vs 前一週」。
只有問句裡明確出現「比較」、「vs」、「對比」等關鍵字才觸發 analysis_compare。
避免 AI 過度解讀使用者意圖，增加不必要的資訊量。

### 為什麼週期日期計算在 server 端而不依賴 GPT？
GPT 解析「上週」的日期範圍不穩定，會因為描述方式不同而產生偏差（如從週日還是週一開始算）。
在 server 端直接用 JS 計算（以本週週一往回推7天），確保「上週」永遠是最近完整的週一到週日，結果可預期且一致。

## 類別設計

### 為什麼自訂類別 icon 用 Lucide 而不是自己畫 SVG？
Lucide 提供 165+ 個語意清晰的 icon，涵蓋所有常見消費類別。
自己畫 SVG 工作量大且風格難統一。
Lucide Vue Next 支援動態渲染（`:is` 綁定），可以用字串名稱直接渲染對應 icon，完美符合「把 icon 名稱存到資料庫」的架構。

### 為什麼類別顏色用莫蘭迪色票？
記帳 app 的資料會同時顯示多個類別，需要一組視覺上和諧不衝突的顏色。
莫蘭迪色系（低飽和度、中高亮度）整體柔和，與 app 的暖橙漸層背景相容。
預先定義 66 個色票，避免 AI 生成的顏色過於鮮豔或過暗。

### 為什麼類別資料要遷移到 Supabase？
初期類別存在 localStorage，導致跨瀏覽器（Chrome/Edge）資料不同步的問題。
記帳資料存在雲端，但類別 icon/顏色卻只在本地，會造成「紀錄有類別名稱但顯示不出 icon」的情況。
遷移到 Supabase 後，同一帳號在任何裝置都能顯示正確的類別 icon，徹底解決同步問題。

## 技術決策

### 為什麼選擇 PWA 而不是原生 App？
目標是快速驗證產品概念，PWA 不需要上架審核，部署即可用。
核心功能（相機、語音、本地儲存）PWA 都能支援。
未來若需要原生功能，可以用 Capacitor 打包成 App，不需要重寫程式碼。

### 為什麼用 Supabase 而不是 Firebase？
Supabase 使用標準 SQL，未來的統計查詢可以直接用 SQL 處理複雜條件。
Firebase 的 NoSQL 結構對「跨時間段、跨類別的聚合查詢」支援較弱。
Supabase 的 Row Level Security 讓使用者資料隔離的實作更直觀。

### 為什麼訪客模式資料合併選擇「localStorage 優先」？
使用者在登入前可能已經在本地修改或新增了類別，這些是比雲端更新的資料。
以 localStorage 為主，確保使用者的最新操作不會被雲端舊資料覆蓋。
合併策略：同名類別以 localStorage 的 icon/color 為主；僅在 localStorage 的類別 insert 到 Supabase。

### 為什麼 CatIcon 元件要同時支援 Lucide 和 SVG path 兩種模式？
預設類別（餐飲、交通等）使用自繪 SVG path，是專案初期就建立的資產。
自訂類別使用 Lucide icon，icon 名稱存在 Supabase。
CatIcon 元件統一處理兩種情況，讓所有頁面只需要傳入 category 名稱，不需要關心 icon 來源。

### 為什麼 SSR 環境下 localStorage 要特別處理？
Nuxt 3 預設啟用 SSR，server 端執行時沒有 `window` 物件，直接讀取 localStorage 會報錯。
解決方式：在 `composables/useUserCategories.ts` 的 setup 階段同步讀取（`process.client` 判斷），確保第一次 render 時就有資料，不需要等 onMounted。

## Design Token 系統

### 為什麼建立 tokens.json 而不是直接用 Tailwind config 或 CSS variables？
prototype 階段 code 裡存在大量不一致的 hardcoded 值（orange.400 和 orange.500 都在當 accent 用、四種透明白、20px 和 22px 的 radius 同時存在）。
單純的 Tailwind config 或 CSS variables 無法表達「為什麼這個值存在」以及「這個值用在哪裡」。
tokens.json 的三層結構（primitive → semantic → component）讓每一個值都有明確的意圖：
- primitive 層：原始值，不帶語意（`color.orange.500 = #E07A4F`）
- semantic 層：用途定義（`color.accent` = 品牌主色，`color.destructive` = 破壞性操作）
- component 層：元件規格（`button-primary.bg` = accent，`card.border-radius` = 24px）

### 為什麼 semantic 字級用情境命名而不是數值命名？
`size-title`、`size-body`、`size-label` 比 `size-16`、`size-14`、`size-12` 更能傳達使用意圖。
當未來需要調整字級時，只需要改 primitive 的數值，所有引用該 semantic token 的地方自動更新。
即使兩個 token 暫時指向相同數值（如 `size-title` 和 `size-input` 都是 16px），保留獨立命名讓未來的差異化調整更容易。

### 為什麼部分值刻意不 tokenize？
Animation keyframes 的顏色是動畫系統的一部分，與 UI token 系統的更新週期不同，混入會造成誤用。
SVG fill 顏色有技術限制（CSS variables 無法直接用於 SVG style 屬性），強制 tokenize 反而增加複雜度。
數據條底色（`#e0ccba`）是視覺設計的刻意選擇，不對應任何 primitive color，獨立存在更清楚。
原則：token 系統應該降低認知負擔，不是把所有值都納管。

### 為什麼先從 code 萃取 token 而不是先在 Figma 調整？
Figma 裡的 frames 是從 Claude Code to Figma 抓進來的，layer 結構不完整，無法可靠地作為數值的來源。
先讓 Claude Code 掃描 code 萃取現有值，得到一份有根據的草稿，再帶著這份草稿去 Figma 做視覺確認和決策。
這個順序讓 Figma 成為「審查和調整」的地方，而不是「挖掘數值」的地方，減少兩端猜測的來回。
